## 第 12 页

项目以资料处理、跨应用协作和研究成果交付为主，Agent 加载 Skill 后完成从资料获取、分析判断、工具调用到报告生成的全流程任务。相较面向软件开发的 Code Agent，Work 类 Agent 更贴合信用研究场景，可更直接地处理长文档、调用办公应用并交付研究成果，减少工作流适配和工程化改造。

## 第 13 页

这一页展示资料收集、文档解析和结构化信息提取环节。按照 OpenAI 官方机制，Web Search 可以执行 search、open_page 和 find_in_page，分别完成检索、打开页面和页内定位，并返回检索来源与引用信息。项目中的 Skill 将发行人名称和证券标识、报告期、资料类型及风险主题转化为检索任务，并定义资料来源优先级，优先采用公司官网、交易所和监管机构、评级机构等权威来源，再补充行业研究和市场信息。资料获取后，Agent 对网页、PDF 和表格进行解析，通过字段映射提取关键事实，对重要数据和判断进行多来源比对，并将缺失或冲突信息标记为待核验事项。左侧展示资料来源分层，右侧视频演示实际的 Web Search 和来源打开过程。

参考：<https://developers.openai.com/api/docs/guides/tools-web-search>

## 第 14 页

这一页展示财务数据处理与定量评分。Agent 从财务报表及附注中提取历史数据，并按照 Skill 预设的口径完成数据标准化；随后计算盈利能力、杠杆、现金流覆盖和流动性等信用分析指标，再依据打分卡中的阈值与映射规则生成定量评分。左侧视频展示了 Agent 执行财务分析和计算任务的实际过程。

## 第 16 页

这一页展示质量检查与交付机制。Agent 在各 Task 结束以及最终交付前，按照检查清单执行规则校验、脚本校验和多模态视觉校验，重点检查主体口径、数据引用、计算结果、评分逻辑和报告格式。检查结果与阶段输出一并留存，用于追溯信息来源、计算过程和评分依据；同时通过统一语言规范、图表模板和渲染规则，提升报告的可读性与版式一致性。

项目中的具体例子包括三类：第一，Task 1 输出前检查文件命名、六个必填部分、资料来源链接，以及是否残留“待补充”或流程性表述。第二，Task 3 对发行人、评级日期、历史审计年份、T-3 至 T+1 期间、币种与单位、字段映射和计算公式逐项校验，并检查定量评分是否正确应用预设阈值。第三，最终报告交付前检查 HTML、PDF 和全页 PNG 预览是否生成，PDF 是否能够正常打开、页面尺寸和字体是否正常、目录与页码是否对应，并通过页面图像检查表格和图表是否存在截断、溢出或空白页。这三类检查分别对应规则约束、脚本校验和多模态视觉校验。

## 第 18 页

这一页概括项目开发过程中遇到的三类主要问题。第一类是“Lost-in-the-Middle”长流程专注力问题：长任务中的部分信息可能被忽略，造成事实遗漏或口径漂移，因此采用 Task 拆分、One Task at a Time、阶段输出落盘、摘要节点、Subagent 和上下文压缩。第二类是输出稳定性：相同任务在字段覆盖、工具选择和输出结构上可能不一致，因此通过示例约束、结构化模板、脚本校验和多模态视觉校验提高一致性。第三类是事实性错误与无依据补全：通过来源引用、多来源核验、覆盖度检查和缺失信息标记进行控制，并明确要求 Agent 承认资料缺失，而不是自行编造。右侧进一步总结了信息提取、写作风格和渲染交付方面形成的项目规则。

## 第 19 页

项目根据任务的持续范围使用两类 Subagent。专项任务型 Subagent 由单个 Task 按需调用，完成资料核验、事件研究、财务分析和图表制作等边界明确的任务；持续协作型 Subagent 跨 Task 保存研究状态，并按需续接和更新结果。这里的“持续协作”不是指 Subagent 始终在后台运行，而是指其研究状态和阶段结果能够跨 Task 保存、读取和复用。

以同业研究为例，主 Agent 将不同可比公司分别交由独立 Subagent 研究，形成结构化事实、比较指标、结论和来源。这些结果在公司研究、业务评分、评级判断和报告组装阶段持续复用，必要时重新唤起 Subagent 补充研究。本页右上角视频展示多个 Subagent 的实际调用过程，右下角表格展示同业研究结果被汇总为标准化比较表。无论采用哪一类方式，主 Agent 都负责任务分派、结果汇总和一致性核验。

## 第 20 页

Task 拆分可以降低单次任务复杂度，但长流程中累计资料仍可能造成信息遗漏、口径漂移和重复读取，仍难以充分解决“Lost-in-the-Middle”问题。项目采用阶段性输出和恢复锚点管理长流程状态：每个 Task 完成后，将研究结论、数据表格、来源和待核验事项写入独立文件，并更新 CURRENT_RATING_STATE.md；后续 Task 按需读取关键结果，从而减少无效上下文占用，并在任务中断后准确恢复执行位置。

恢复锚点只记录当前进度、下一步任务和已完成输出路径，不承载研究结论，也不作为报告资料来源。右侧截图展示了实际运行中的上下文压缩提示、Subagent 列表及结构化研究输出。

## 第 21 页

这一页将多种重复问题归纳为“输出与交付规范不一致”：Agent 能够完成任务，但固定字段可能混入解释或判断过程，数据缺失时可能使用不统一的表述，图表标题可能只描述内容，表格和图表样式也可能偏离模板。

项目在 Skill 中同时使用明确规则、One-shot / Few-shot 示例和自动校验。固定字段、缺失值等格式相对固定的输出，可以通过一个标准样例明确填写方式；图表标题、研究语言等存在多种场景的表达，则需要多个代表性示例。结构化模板、脚本和视觉检查负责可机械验证的内容。

下方三张图是实际应用。左图以首次评级报告的固定字段为标准样例，约束字段内容和缺失值写法；中间和右侧分别展示业务结构与现金流图表，Skill 通过多组标题示例，让 Agent 学习在标题中同时写出事实判断及信用含义，例如业务和地域分散程度、债务覆盖能力与资本开支压力。
